A confusion-cluster classifier has already mapped the student's typed phrase to a known cluster. Your job is to produce a sub-simulation that directly resolves that cluster's confusion in the minimum number of states possible.

Concept: {CONCEPT_ID} — {CONCEPT_NAME}

{PARENT_SCENE_COMPOSITION}

{REAL_WORLD_ANCHOR}

{PHYSICS_ENGINE_CONFIG}

cluster_id: {CLUSTER_ID}

{CLUSTER_TRIGGER_EXAMPLES}

PROTOCOL SELECTION
==================
Pick ONE:
- MICRO (exactly 2 states): When the confusion is about a single symbol, term, or direction.
- LOCAL (2–4 states): When the confusion needs a small derivation or a chain of 2–3 ideas.

YOUR TASK
=========
Produce the sub-simulation. Every state:

1. `scene_composition`: 3+ primitives (surface, body, force_arrow, force_components, vector, annotation, formula_box, angle_arc, label, motion_path). Do NOT use `comparison_panel` — renderer limitation: only 2 sub-scenes supported, 3 leaves right panel empty.
2. `focal_primitive_id`: one primitive id in the scene.
3. `teacher_script.tts_sentences`: 2–4 sentences, plain English, Indian context, no Hinglish. Each with `id` + `text_en`.
4. `advance_mode`: mix of `auto_after_tts`, `manual_click`, `wait_for_answer`, `interaction_complete`. If MICRO protocol and 2 states, pick distinct modes.

LAYOUT VOCABULARY — CRITICAL (new in v2)
========================================
You MUST NOT emit absolute `position: { x, y }` pixel coordinates for `label`, `annotation`, or `formula_box` primitives. A constraint solver resolves positions at render time from the relationships you describe. Use the grammar below.

Required fields on every `label`, `annotation`, `formula_box`:
- `anchor` (required): attach-point name. Examples: "block.top_center", "MAIN_ZONE.slot_1", "floor.mid", "on_surface:incline at:0.35".
- `edge` ("top" | "bottom" | "left" | "right", default "top"): which side of the anchor the primitive sits on.
- `gap` (0–200, default 12): pixel distance between anchor and primitive edge.
- `priority` ("required" | "strong" | "medium" | "weak", default "strong").

Anchor grammar:
- Zone: `MAIN_ZONE.center`, `CALLOUT_ZONE_R.slot_1`, `FORMULA_ZONE.center`, etc.
- Body: `<body_id>.top`, `.bottom`, `.left`, `.right`, `.center`, `.top_center`, `.bottom_center`.
- Surface: `<surface_id>.start`, `.mid`, `.end`, or `on_surface:<surface_id> at:<fraction>`.

Worked example — a label naming the component of mg that drives motion down the incline:
```
{
  "type": "label",
  "id": "mg_parallel_dr2",
  "text": "mg sin θ",
  "font_size": 16,
  "anchor": "block.center",
  "edge": "bottom",
  "gap": 18,
  "priority": "strong",
  "appear_at_ms": 1800
}
```

Priority rule (CRITICAL — avoids "Anchor tie failed: unsatisfiable constraint" warnings):
Use `"priority": "strong"` on EVERY label, annotation, and formula_box by default. `"required"` tells the solver the anchor MUST snap, even past the canvas edge — if a long formula centered near FORMULA_ZONE.top_center exceeds the 760 px canvas width, kiwi throws and the primitive drops to (0, 0). `"strong"` lets the solver relax gracefully. Reserve `"required"` for at most ONE tiny label per state where pixel-perfect snap is mandatory (rare).

Keep formula_box `equation_expr` short (≤ 30 characters, ideally ≤ 25). FORMULA_ZONE is only 255 pixels wide; long strings like "Perpendicular: mg cos θ → balanced by N" will not fit and WILL break layout. Split sentences into `annotation` primitives in CALLOUT_ZONE_R — formulas are for equations, not narration.

FORMULA_ZONE stacking rule (CRITICAL — avoids solver "unsatisfiable constraint" warnings):
If your drill-down emits 2+ formula_boxes in one state, NEVER reuse the same (anchor, edge) pair. Distribute across the three tiers:
- 1st: `"anchor": "FORMULA_ZONE.top_center", "edge": "bottom", "gap": 6`
- 2nd: `"anchor": "FORMULA_ZONE.center", "edge": "top", "gap": 0`
- 3rd: `"anchor": "FORMULA_ZONE.bottom_center", "edge": "top", "gap": 6`
Same applies to multiple labels on one body — use different `edge` values or different body anchor points.

Surface geometry rule (CRITICAL — prevents off-canvas walls):
- Horizontal floor: `{"orientation": "horizontal", "position": {"x": 80, "y": 420}, "length": 400}`.
- Inclined ramp (0 < θ < 90): `{"orientation": "inclined", "position": {"x": 80, "y": 420}, "angle_expr": "30", "length": 380}` — starts at lower-left and runs up-and-right.
- Vertical wall: use `"orientation": "vertical"` with `"position": {"x": 300, "y": 430}` and `"length": 330`. Do NOT express a vertical wall as `orientation: "inclined" angle: 90` with `y: 100` — the server-side registry computes an endpoint at y = –230 (off-canvas top) and every body attached to that surface renders off-screen. A 90° demo is better staged as `angle: 89` on an inclined surface so the perpendicular direction stays numerically stable.

FORBIDDEN on label/annotation/formula_box:
- `"position": { "x": ..., "y": ... }` — do not emit. The solver will ignore it anyway.
- Anchor strings that do not match the grammar above.
- Curly-brace template variables in text fields (e.g. `{N_value}`, `{mg_parallel}`). Write the actual value or symbolic expression instead. The renderer cannot substitute invented variable names — they appear verbatim on screen.

Force_arrow schema (NOT solver-managed — use these fields):
```
{
  "type": "force_arrow",
  "id": "N_arrow_dr1",
  "from": "block.top_center",
  "magnitude": 20,
  "direction_deg": 90,
  "scale_pixels_per_unit": 4,
  "color": "#10B981",
  "label": "N"
}
```
- `from` uses the SAME anchor grammar as `anchor`.
- `direction_deg`: 0=right, 90=up, 180=left, 270=down.
- Force_arrows NEVER carry `anchor`/`edge`/`gap`/`position` — omit them. Route through `from` + `direction_deg`.

Surfaces, bodies, vectors, force_components, angle_arcs, motion_paths, comparison_panels continue to use their existing fields. The solver only owns label/annotation/formula_box layout — every other type must carry its own positioning field.

Pedagogy:
- STATE_1 must directly mirror the student's wrong belief or stuck point. Name the confusion out loud so the student feels understood.
- Final state must resolve the specific cluster confusion — not a generic re-explanation.
- Keep it tight: students typed one confusion; don't balloon into a mini-lesson.
- Give each primitive a stable `id` suffixed `_dr{N}` (e.g. `mg_component_dr1`) to avoid collisions with the parent state.
