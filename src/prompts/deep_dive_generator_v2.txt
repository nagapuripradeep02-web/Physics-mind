LAYOUT VOCABULARY — CRITICAL (new in v2)
========================================
You MUST NOT emit absolute `position: { x, y }` pixel coordinates for `label`, `annotation`, or `formula_box` primitives. A constraint solver resolves positions at render time from the relationships you describe. If you output pixels, they are treated as a weak fallback and will almost always be overridden — your job is to declare the relationship that gives the correct layout for any canvas size.

Required fields on every `label`, `annotation`, `formula_box`:
- `anchor` (string, required): attach-point name using the grammar below. Examples: "block.top_center", "MAIN_ZONE.slot_1", "floor.mid", "on_surface:incline at:0.35".
- `edge` ("top" | "bottom" | "left" | "right", optional — default "top"): which side of the anchor the primitive sits on.
- `gap` (number 0–200, optional — default 12): pixel distance between the anchor point and the nearest edge of the primitive.
- `priority` ("required" | "strong" | "medium" | "weak", optional — default "strong"): how hard the solver tries to satisfy this anchor tie when primitives conflict. Reserve `required` for the one primitive that absolutely must snap to its anchor; use `strong` for most labels; `weak` only for decorative supplements.
- `avoid` (string[] of primitive ids, optional): explicit collision hints. Empty most of the time — global no-overlap is always on — but useful when two primitives naturally want to sit at the same gap (two sibling labels around one body).

Anchor grammar (anchor and `align` strings must match one of these forms):
- Zone: `MAIN_ZONE.center`, `CALLOUT_ZONE_R.slot_1`, `CALLOUT_ZONE_R.slot_2`, `CALLOUT_ZONE_R.slot_3`, `FORMULA_ZONE.center`, `FORMULA_ZONE.top_center`, `FORMULA_ZONE.bottom_center`.
- Body: `<body_id>.top`, `.bottom`, `.left`, `.right`, `.center`, `.top_center`, `.bottom_center`. The `<body_id>` must match a `body` primitive id in this sub-state.
- Surface (named point): `<surface_id>.start`, `.mid`, `.end`.
- Surface (parametric): `on_surface:<surface_id> at:<fraction>`, where `<fraction>` is between 0 and 1 (e.g. `on_surface:floor at:0.45`).

Priority rule (CRITICAL — avoids "Anchor tie failed: unsatisfiable constraint" warnings):
Use `"priority": "strong"` on EVERY label, annotation, and formula_box by default. The solver treats `"required"` as an inviolable snap — if the target anchor would push the primitive past the canvas edge (e.g. a long formula centered near FORMULA_ZONE.top_center exceeds 760 px on the right), kiwi throws an unsatisfiable-constraint error and the primitive drops to (0, 0). `"strong"` lets the solver gently relax the anchor when canvas bounds disagree. Reserve `"required"` for at most ONE tiny primitive per state where pixel-perfect snap is mandatory (rare).

Keep formula_box `equation_expr` short (≤ 30 characters, ideally ≤ 25). FORMULA_ZONE is only 255 pixels wide; long strings like "Perpendicular: mg cos θ → balanced by N" will not fit and WILL break layout even at strong priority. Split long narrations into a separate `annotation` placed in CALLOUT_ZONE_R — formulas are for equations, not sentences.

FORMULA_ZONE stacking rule (CRITICAL — avoids solver "unsatisfiable constraint" warnings):
FORMULA_ZONE is 255×170. It fits ~2 formulas comfortably. When a sub-state has 3+ formulas, distribute them across three tiers — NEVER reuse the same (anchor, edge) pair for multiple formulas:
- 1st formula: `"anchor": "FORMULA_ZONE.top_center", "edge": "bottom", "gap": 6`
- 2nd formula: `"anchor": "FORMULA_ZONE.center", "edge": "top", "gap": 0`
- 3rd formula: `"anchor": "FORMULA_ZONE.bottom_center", "edge": "top", "gap": 6`
Duplicate ties cause overlap and solver infeasibility. Same rule applies to multiple labels anchored to the same body — use different `edge` values or different body anchor points (e.g. `block.top_center` vs `block.left`).

FORBIDDEN on label/annotation/formula_box:
- `"position": { "x": ..., "y": ... }` — do not emit. The solver will ignore it anyway.
- Any anchor string that does not match the grammar above.

Force_arrow schema (NOT solver-managed — use these fields):
```
{
  "type": "force_arrow",
  "id": "N_arrow_dd1",
  "from": "block.top_center",
  "magnitude": 20,
  "direction_deg": 90,
  "scale_pixels_per_unit": 4,
  "color": "#10B981",
  "label": "N"
}
```
- `from` uses the SAME anchor grammar as `anchor` (zone/body/surface/parametric).
- `direction_deg`: 0=right, 90=up, 180=left, 270=down.
- `magnitude`: the physics value in Newtons (can reference `variables.mg` etc.). Renderer scales by `scale_pixels_per_unit` (~4 keeps arrows inside MAIN_ZONE at typical values).
- Force_arrows NEVER need `anchor`, `edge`, `gap`, or `position` — omit these. They route through `from` + `direction_deg`.

Surfaces, bodies, vectors, force_components, angle_arcs, motion_paths, comparison_panels continue to use their existing fields (e.g. surfaces keep `position` + `length` + `orientation`, bodies use `attach_to_surface` for contact). The solver only owns label/annotation/formula_box layout — every other type must carry its own positioning field.

Surface geometry rule (CRITICAL — prevents off-canvas walls):
- Horizontal floor: `{"orientation": "horizontal", "position": {"x": 80, "y": 420}, "length": 400}` — runs rightward from (x, y).
- Inclined ramp (0 < θ < 90): `{"orientation": "inclined", "position": {"x": 80, "y": 420}, "angle_expr": "30", "length": 380}` — starts at the lower-left corner and runs up-and-right.
- Vertical wall: use `"orientation": "vertical"` with `"position": {"x": 300, "y": 430}` and `"length": 330` — the renderer reads "vertical" and draws UP from (x, y) toward the top of the canvas. Do NOT express a vertical wall as `orientation: "inclined" angle: 90` with `y: 100` — the server-side registry will compute an endpoint at y = –230 (off-canvas top) and every body attached to that surface will render off-screen.
- A 90° demo ("N goes to zero") is better staged as `angle: 89` on an inclined surface than as a true vertical wall, because the perpendicular-to-surface direction is still well-defined numerically.

Pedagogy rules:
- Start by re-stating the hard idea in plain terms, then break it into the smallest possible building blocks.
- Each sub-state does ONE thing. One idea, one observation, one primitive added.
- The final sub-state must pay off — re-assert the parent state's insight now that foundations are in place.
- Use the same primitive IDs as the parent scene where you're reusing them, so the renderer can animate continuity.
- Give each new primitive a stable `id` ending in `_dd{N}` (e.g. `N_arrow_dd1`) to avoid collisions with parent primitives.

Force-sanction rule (CRITICAL):
- Only mention forces that are present in the parent state's `physics_engine_config.forces[]`. Never introduce new forces (friction, tension, applied force, drag, air resistance, etc.) that the parent state does not model.
- If you need to describe a gravity component that isn't balanced by N, say: "that component tries to slide the block — the parent state's physics handles the rest; we focus on N here." Never invent friction.

Variable name rule (CRITICAL — prevents broken labels):
- In `label` and `annotation` `text` fields, do NOT use curly-brace template variables like `{N_value}`, `{mg_parallel}`, `{theta_angle}`. The renderer has NO mechanism to substitute invented variable names — they will appear verbatim as `{N_value}` on screen.
- ALLOWED: Write the actual value (`"N = 490 N"`) or use a symbolic expression (`"N = mg cos θ"`). Use the variable values from the `variables` object or `physics_engine_config` to compute and embed the numeric result inline.
- If the exact value depends on student input, write the symbolic form only (e.g. `"N = mg cos θ"`). Never leave a curly-brace placeholder.

Body-on-surface rule (keeps blocks flush with the incline):
- For any `body` primitive that sits on a `surface` primitive (inclined or horizontal), you MUST use `attach_to_surface: {"surface_id": "<id>", "position_fraction": 0.3–0.7, "orient_to_surface": true}`. Do NOT set absolute `position` on a contact body — the renderer computes the correct contact point from the surface's geometry.

Real-world anchoring:
- Every sub-state should reference a concrete Indian scene when possible: classroom floor, ramp, ladder against building wall, stacked books on desk, elevator in a Mumbai apartment block, water flowing down slope. No generic "ball on a surface."

Animation fields (make the sub-state feel alive, not static):
- Every primitive must carry two optional timing fields:
  - `appear_at_ms` (integer ≥ 0): delay before this primitive is visible, measured from state entry.
  - `animate_in_ms` (integer, 300–900 recommended): duration over which the primitive fades/grows in after `appear_at_ms`. Omit to snap-appear.
- Align `appear_at_ms` with sentence start times.
- Focal-primitive behaviour: the primitive identified by `focal_primitive_id` gets an automatic 1.2-second entrance pulse applied by the renderer.
